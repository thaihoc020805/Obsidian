The third type is a <mark style="background: #FFF3A3A6;">LoadBalancer</mark>, where it provisions a load balancer for our service in supported cloud providers – like Google Cloud, AWS or Azure. <mark style="background: #BBFABBA6;">A good example of that would be to distribute load across different web servers</mark>. <mark style="background: #FFF3A3A6;">A nodeport is a type of service where a normal service is created first. And is then exposed to external users through a port on the node</mark>. Let’s take a closer look at the Service. If you look at it,<mark style="background: #BBFABBA6;"> there are 3 ports involved.</mark> <mark style="background: #FFF3A3A6;">The port on the POD were the actual web server is running is port 80</mark>. Let us now look at <mark style="background: #FFF3A3A6;">how to create the service</mark>. As before we will use a definition file to create a service. The <mark style="background: #FFF3A3A6;">high level structure of the file remains the same</mark>. we have apiVersion, kind, metadata and spec sections. The apiVersion is going to be v1. The kind is ofcourse service. The metadata will have a name and that will be the name of the service. It can have labels, but we don’t need that for now. Next we have <mark style="background: #FFF3A3A6;">spec</mark>. and as always this is the most crucial part of the file as this is were we will be <mark style="background: #FF5582A6;">defining the actual services and this is the part of a definition file that differs between different objects</mark>. In the spec section of a service we have<mark style="background: #FFF3A3A6;"> type and ports</mark>. The type refers to the type of service we are creating. As discussed before it could be <mark style="background: #FFF3A3A6;">ClusterIP, NodePort, or LoadBalancer</mark>. In this case since we are creating a <mark style="background: #FFF3A3A6;">NodePort </mark>we will set it as NodePort. The next part of spec is ports. This is where we input information regarding what we discussed on the left side of this screen. <mark style="background: #FFF3A3A6;">The first type of port is the targetPort, which we will set to 80. The next one is simply port, which is the port on the service object and we will set that to 80 as well. The third is NodePort which we will set to 30008 or any number in the valid range</mark>. Remember that out of these,<mark style="background: #FF5582A6;"> the only mandatory field is port. If you don’t provide a targetPort it is assumed to be the same as port and if you don’t provide a nodePort a free port in the valid range between 30000 and 32767 is automatically
allocated</mark>. 
Also note that<mark style="background: #FF5582A6;"> ports is an array</mark>. So note the dash under the ports section that indicate the ﬁrst element in the array. <mark style="background: #BBFABBA6;">You can have multiple such port mappings within a single service</mark>.

So we have all the information in, but something is really missing. <mark style="background: #FFF3A3A6;">There is nothing here in the deﬁnition ﬁle that connects the service to the POD</mark>. We have simply speciﬁed the targetPort but we didn’t mention the targetPort on which POD. T<mark style="background: #BBFABBA6;">here could be 100s of other PODs with web services running on port 80</mark>. So how do we do that? As we did with the replicasets previously and a technique that you will see very oqen in kubernetes, we will use l<mark style="background: #BBFABBA6;">abels and selectors to link these together</mark>. We know that the POD was created with a label. <mark style="background: #FFF3A3A6;">We need to bring that label into this service deﬁnition ﬁle</mark>.


![[Pasted image 20250803003153.png]]

So we have a new property in the spec section and that is **selector**. Under the selector, <mark style="background: #FFF3A3A6;">provide a list of labels to identify the POD</mark>. For this, refer to the pod-definition file used to create the POD. Pull the labels from the pod-definition file and place it under the selector section. This links the service to the pod. Once done, create the service using the <mark style="background: #FFF3A3A6;">`kubectl create` command</mark> and input the <mark style="background: #FFF3A3A6;">service-definition file</mark> and there you have the service created. To see the created service, run the <mark style="background: #FFF3A3A6;">`kubectl get services` </mark>command that lists the services, their <mark style="background: #FFF3A3A6;">cluster-ip</mark> and <mark style="background: #FFF3A3A6;">the mapped ports</mark>. <mark style="background: #BBFABBA6;">The type is NodePort as we created and the port on the node automatically assigned is 32432. We can now use this port to access the web service using curl or a web browser</mark>.

![[Pasted image 20250803004019.png]]

So far we talked about a service mapped to a single POD. But that’s not the case all the time, what do you do <mark style="background: #FFF3A3A6;">when you have multiple PODs?</mark> In a production environment you have multiple instances of your web application running for high-availability and load balancing purposes. In this case we have<mark style="background: #FFF3A3A6;"> multiple similar PODs running our web application.</mark> <mark style="background: #BBFABBA6;">They all have the same labels with a key name set to redis-pod</mark>. The <mark style="background: #BBFABBA6;">same label is used as a selector during the creation of the service</mark>. So <mark style="background: #FFF3A3A6;">when the service is created, it looks for matching PODs with the labels and finds 3 of them</mark>. <mark style="background: #BBFABBA6;">The service then automatically selects all the 3 PODs as endpoints to forward the external requests coming from the user</mark>. <mark style="background: #BBFABBA6;">You don’t have to do any additional configuration to make this happen.</mark> And if you are wondering what algorithm it uses to balance load,<mark style="background: #FF5582A6;"> it uses a random algorithm</mark>. <mark style="background: #BBFABBA6;">Thus, the service acts as a built-in load balancer to distribute load across different PODs.</mark>

![[Pasted image 20250803004330.png]]

And finally, l<mark style="background: #BBFABBA6;">et's look at what happens when the PODs are distributed across multiple nodes</mark>. In this case, we have the <mark style="background: #FFF3A3A6;">web application on PODs on separate nodes in the cluster</mark>. <mark style="background: #FFF3A3A6;">When we create a service</mark>, without us having to do ANY kind of additional configuration, <mark style="background: #BBFABBA6;">Kubernetes creates a service that spans across all the nodes in the cluster and maps the target port to the SAME NodePort on all the nodes in the cluster</mark>. This way <mark style="background: #FF5582A6;">you can access your application using the IP of any node in the cluster and using the same port number which in this case is 30008</mark>. To summarize – <mark style="background: #FFF3A3A6;">in ANY case whether it be a single pod in a single node, multiple pods on a single node, multiple pods on multiple nodes, the service is created exactly the same without you having to do any additional steps during the service creation</mark>. <mark style="background: #BBFABBA6;">When PODs are removed or added the service is automatically updated making it highly adaptive</mark>. Once created you won’t typically have to make any additional conﬁguration changes.
